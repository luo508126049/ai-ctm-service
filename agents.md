# AI 编程规范

本文档约束 AI Agent 在本项目中的基础编程行为。所有自动生成、修改、重构代码的操作都必须遵守本规范。

## 基本原则

- 先理解现有代码，再进行修改。
- 优先沿用项目已有技术栈、目录结构、命名方式和编码风格。
- 不为单次需求引入不必要的新框架、新中间件或复杂抽象。
- 修改范围必须和用户需求直接相关，避免顺手重构无关代码。
- 不删除用户已有改动，除非用户明确要求。
- 不提交真实密钥、Token、账号密码或内部地址。
- 面向可运行结果交付，修改后必须尽量执行编译、测试或启动验证。

## 架构边界

- Controller 层只做 HTTP 协议适配、参数接收和结果返回。
- Service 层承载业务编排、AI 调用编排和事务边界。
- Tool 层承载 AI 可调用工具入口，不直接膨胀为复杂业务实现。
- 配置放在 `application*.yaml` 或环境变量中，不在 Java 代码中硬编码环境差异。
- 前端静态资源放在 `src/main/resources/static`，后端接口放在 `src/main/java`。
- 脚本类工具放在 `scripts/` 目录，运行时产物放在 `.run/`。

## Java 编程规范

- 使用 Java 17 语法特性时保持可读性，不为了炫技使用复杂写法。
- 使用构造器注入，禁止字段注入。
- 公共类和公共方法应提供必要 Javadoc，说明职责、参数和返回值。
- 日志使用 SLF4J，不使用 `System.out.println`。
- 异常不能吞掉；捕获异常后必须记录、转换或继续抛出。
- DTO、record、枚举应表达稳定的业务含义。
- 不在 Controller 中直接访问数据库、外部 HTTP 服务或大模型客户端。

## Spring AI 规范

- 统一通过 Spring AI `ChatClient` 接入对话模型。
- 模型提供方通过 Spring Profile 切换，不在业务代码中写 if/else 判断模型类型。
- 本地开发默认使用 `local` profile 和 DeepSeek。
- 无互联网服务器使用 `server` profile 和 Ollama。
- `@Tool` 方法名必须稳定，避免频繁改名导致提示词和调用链失效。
- `@Tool` 描述必须说明触发场景、输入含义和行为边界。
- 写操作工具必须明确区分“查询”和“变更”，不得让模型隐式执行高风险操作。
- 工具返回值优先使用 record 或明确 DTO，不返回松散字符串。

## 配置规范

- 本地敏感配置优先通过环境变量提供。
- `application-local.yaml` 只放本地开发默认配置。
- `application-server.yaml` 只放服务器部署默认配置。
- 不在 README、代码注释或测试样例中写真实生产密钥。
- 新增配置项必须说明默认值、环境变量名和适用 profile。

## 前端规范

- 保持原生静态前端轻量结构，除非需求明确，不引入前端构建链。
- 与后端交互时复用现有 `/api/chat` 和 `/api/chat-stream` 接口约定。
- UI 文案应面向最终用户，避免暴露内部类名、工具名或技术细节。
- SSE 交互必须处理连接失败、用户停止、服务端完成三类状态。

## 测试与验证

- 后端改动后至少执行 `.\mvnw.cmd test`。
- Profile 配置改动后分别验证 `local` 和 `server` 容器加载。
- AI Tool 改动后必须做至少一次自然语言触发烟测。
- 启动脚本改动后必须验证 `start`、`status`，必要时验证 `stop` 或 `restart`。
- 文档改动不强制跑测试，但必须确保命令、路径、端口和 profile 名称与项目一致。

## 文档规范

- README 面向项目使用者，说明项目定位、启动方式、配置和验证方法。
- `agents.md` 面向 AI Agent，说明项目内编码边界和约束。
- 新增重要能力时同步更新 README。
- 新增 AI 编程边界或约束时同步更新 `agents.md`。

## 禁止事项

- 禁止擅自执行破坏性 Git 操作，例如 `git reset --hard`、强制 checkout。
- 禁止删除或覆盖用户未提交改动。
- 禁止在未验证的情况下声称功能可用。
- 禁止把示例数据描述成真实生产数据。
- 禁止将模型供应商、模型名称、端口、路径硬编码在业务代码中。
- 禁止为通过编译而移除核心业务逻辑。
